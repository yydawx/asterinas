## Context

当前的 `BoundPort::endpoint()` 方法在 `kernel/libs/aster-bigtcp/src/iface/common.rs` 中仅返回 IPv4 地址（如果可用）。然而，在 IPv6 支持被添加之后（提交 0046aba7），此方法被修改为优先检查 IPv6 地址，然后才是 IPv4 地址。

问题在于：一个套接字只能绑定到一个特定的接口（要么是 IPv4，要么是 IPv6），但 `endpoint()` 方法可能会返回与实际绑定接口不匹配的地址族。例如：
- 一个 socket 绑定到仅有 IPv4 地址的接口
- 但是，如果系统上存在其他带有 IPv6 地址的接口，endpoint() 可能会返回那个 IPv6 地址
- 这导致 ConnectionKey 使用了错误的本地 IP 地址

## Goals / Non-Goals

**Goals:**
- 修正 `BoundPort::endpoint()` 使其返回与实际绑定接口匹配的地址族
- 确保 IPv4 和 IPv6 socket 的行为一致
- 维持与现有 API 的向后兼容性（无公开更改）

**Non-Goals:**
- 改变端口分配或绑定逻辑
- 修改 used_ports 跟踪机制
- 改变套接字创建或系统调用处理

## Decisions

### 决策点：如何确定正确的地址族
考虑了几种方法：

1. **检查接口上可用的地址**（当前方法）
   - 问题：如果接口同时具有 IPv4 和 IPv6 地址，哪个应该优先？
   - 问题：这仍然无法知道套接字实际绑定到了哪个地址

2. **在绑定时存储地址族信息**
   - 在 BoundPort 结构中添加一个字段来记录地址族
   - 优点：精确、直接
   - 缺点：需要修改 BoundPort 结构并更新所有使用者

3. **根据远程地址推断地址族**（用于连接套接字）
   - 仅适用于已建立连接的套接字
   - 对于监听套接字或未连接的数据报套接字无效

4. **查询底层 smoltcp 套接字**（最准确但最复杂）
   - 需要访问内部 smoltcp 套接字状态
   - 复杂性高，可能影响性能

**所选择的方法**: 决策点：如何确定正确的地址族
我们选择了方案 #2：在 BoundPort 结构中存储地址族信息。

理由：
- 这是最准确的方法 - 我们知道绑定时使用了什么地址
- 只需要对 BoundPort 结构进行最小的修改
- 使得 endpoint() 方法变得简单直接
- 不依赖于可能在绑定后发生变化的外部状态
- 对所有套接字类型（监听、连接、数据报）均有效

### 替代方案考虑
对于决策点：如何确定正确的地址族：

**方案 #1（检查接口地址）被拒绝**因为：
- 它无法区分当同一接口上同时存在IPv4和IPv6地址时应该使用哪个
- 它依赖于当前接口状态，这可能在绑定之后发生变化
- 它不能保证返回的地址实际上被套接字使用

**方案 #3（基于远程地址推断）被拒绝**因为：
- 它仅适用于已建立连接的套接字
- 监听套接字没有远程地址可用
- 数据报套接字在发送之前可能没有远程地址

**方案 #4（查询smoltcp套接字）被拒绝**因为：
- 它需要暴露更多的smoltcp内部信息
- 它增加了在热路径上的复杂性
- 当我们在绑定时已经知道答案时，这是不必要的

## 风险 / 权衡

[内存使用增加] → BoundPort 结构将增加一个字段来存储地址族信息（大约每个绑定端口增加 1-2 字节）

[需要更新所有 BoundPort 创建点] → 必须修改 IfaceCommon::bind() 方法以捕获并存储地址族信息

[在过渡期间可能存在不一致] → 现有的 BoundPort 实例将需要被替换（通过自然的套接字生命周期）

## 迁移计划

1. 修改 BoundPort 结构以添加 address_family 字段
2. 更新 IfaceCommon::bind() 方法以在创建 BoundPort 时捕获地址族
3. 修改 BoundPort::endpoint() 方法以使用存储的地址族而不是检查接口
4. 确保在绑定过程中正确识别地址族（IPv4 vs IPv6）
5. 运行现有的测试以确保没有回归
6. 添加特定的测试来验证 IPv4 和 IPv6 绑定场景

## 未解决的问题

1. BoundPort 应该存储具体的 IP 地址还是仅仅存储地址族？
   - 存储地址族较为简单，但如果我们需要实际的IP地址用于endpoint()，可能需要存储完整地址
   - 不过，我们仍然需要从接口获取实际的IP地址，所以存储地址族应该就足够了

2. 如何处理同时具有IPv4和IPv6地址的接口？
   - 地址族应该反映用于绑定的地址类型，这由传递给bind_port的endpoint决定
   - IfaceCommon::bind() 应该检查用于端口分配的endpoint的地址族

3. 这是否会影响性能？
   - 添加一个字段到 BoundPort 中的影响可以忽略不计
   - endpoint() 方法实际上会变得更快，因为它不再需要检查两个地址类型
